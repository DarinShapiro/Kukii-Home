# Epic 10.11: Pluggable, Parallel, Distributed Identity DAG + Soft-Biometric Pipelines

**Parent epic:** Epic 10 (Identity & Recognition) · builds on 10.10 (multi-modal identity & fusion)
**Components:** services/preprocessor (IdentityRouter, EnrolledCorpus, pipelines/identity/\*), shared/lib-python (contracts, topology), services/preprocessor deployment/topology
**Priority:** P1 — the architecture that lets us add clothing-robust biometrics _and_ scale inference across resources without rewriting the router each time
**Builds on:** Epic 10.4 (face), 10.5.1 (body), 10.5.3 (pet), 10.10 (fusion) — all landed

---

## Goal

Three asks, one architecture:

1. **Plug in any model as needed** — adding a new identity modality (CC-ReID, gait, 3D body shape, calibrated height, plate, …) should be _register a pipeline_, with **zero edits** to the router or the corpus.
2. **Run in parallel** — independent pipelines and per-frame work execute concurrently, bounded by real resource limits, not a fixed thread pool.
3. **Distribute across resources** — a model may live on another host; a host may have **multiple GPUs**. Placement is config, and the router treats a remote model identically to a local one.

Plus: **add the soft-biometric pipelines** scoped in 10.10's clothing-robustness discussion (CC-ReID, gait, 3D shape, height) as first-class plugins.

## What exists today (credit the current router)

`IdentityRouter` (`pipelines/identity/router.py`) already does a lot right:

- **Plugin Protocol** (`IdentityPipeline`): `name`, `triggers_on`, `depends_on`, `skip_when_upstream_matched_above`, `has_enrollments`, `run`.
- **DAG**: `_build_branches` topo-sorts pipelines into dep-linked chains (face→body), missing-dep tolerant (register a subset for tests).
- **Concurrency**: independent branches dispatch via `asyncio.gather` per frame; within a branch, sequential with short-circuit (`skip_when_upstream_matched_above`).
- **Gating**: by detection `kind` (`triggers_on`) and by `has_enrollments` — skips JPEG-decode + inference when irrelevant.

Its own docstring lists what's missing: _per-backend semaphores, budget timeout, per-pipeline telemetry._ This epic is those, plus pluggability and distribution.

## Gaps to close

| Ask                           | Gap today                                                                                                                                                                                                                           |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Pluggability**              | `EnrolledCorpus` has **hardcoded** modality fields (`faces`, `bodies`, `pets`, `plates`). A new modality requires editing this class + `from_cache`. Not "register and go."                                                         |
| **Parallel / resource-aware** | `asyncio.gather` over _all_ branches × _all_ frames spawns unbounded executor tasks. No per-device concurrency cap, no batching across frames, no backpressure. Fine for 3 models on one box; breaks with 7 models or a shared GPU. |
| **Distributed / multi-GPU**   | Router assumes **in-process** pipelines. No remote transport, no device/host placement, no multi-GPU pool.                                                                                                                          |
| **New modalities**            | CC-ReID / gait / shape / height pipelines don't exist; gait needs a _sequence_ not a single frame — a shape the current per-frame `run()` doesn't model.                                                                            |
| **Telemetry**                 | No per-pipeline timing (ties to Epic #96 StepTimings).                                                                                                                                                                              |

## Design

### 1. Generalized template registry (pluggability)

Replace the hardcoded `EnrolledCorpus` fields with a **modality-keyed store**:

```python
class EnrolledCorpus:
    templates: dict[str, dict[str, np.ndarray | str]]   # modality -> {actor_id -> template}
    actor_names: dict[str, str]
    def slice(self, modality: str) -> dict[str, ...]: ...
```

- A pipeline declares `modality: str` (e.g. `"face"`, `"body"`, `"gait"`, `"shape"`, `"height"`) and reads `corpus.slice(self.modality)`.
- `ActorEnrollmentEvent` / `KnownActor` carry **template lists per modality** (from 10.10.1) under the same string keys → `from_cache` projects generically, no per-modality `if`.
- **Result:** a new modality = a new pipeline class + a new modality key. Router and corpus untouched.

### 2. Pipeline capability descriptor (placement + scheduling inputs)

Extend the Protocol with declarative resource metadata (all optional, sane defaults):

```python
resource_class: str          # "gpu" | "cpu" | "npu"  — what it wants
batchable: bool              # can it take N crops in one call? (OSNet yes, gait per-track)
temporal: bool               # needs a frame SEQUENCE, not a single frame (gait, SMPL-video)
est_cost_ms: int             # rough per-call cost, for the scheduler
placement_hint: str | None   # logical node/pool name; resolved by topology
```

`temporal=True` pipelines get a new `run_sequence(frames, tracks, corpus)` entry (gait/shape need the track's frame history), leaving the existing per-frame `run()` for the rest.

### 3. Resource-aware scheduler (parallelism done right)

A **`ResourcePool`** layer between the router and pipelines:

- One **semaphore per logical resource** — a 2-GPU node = pool size 2; a single GPU = 1; CPU pool = `min(cores, N)`. Inference acquires a slot before running, releases after.
- **Batching**: for `batchable` pipelines, coalesce a window's crops (across frames) into one inference call per acquire — already how OSNet/DINOv2 want to run (N×crop in one `session.run`). Cuts per-call overhead and GPU dispatch count.
- **Backpressure**: bounded concurrency replaces unbounded `gather`; excess work queues. Keeps a 7-model set from thrashing a shared GPU.
- **Budget/timeout** per `identify()` window: cap wall-clock; drop the slowest non-critical branch and `log()` it (no silent truncation).

The router still topo-sorts branches; the _scheduler_ decides how many run at once, per resource.

### 4. Model placement & distribution

- **`RemotePipeline`** — an `IdentityPipeline` implementation that is an **RPC stub** to another inference node (reuse the FrameWindow transport, or a dedicated identity-RPC over the existing NATS bus / gRPC). To the router it's just a pipeline; `run()` ships the crop(s) + corpus slice ref and awaits matches.
- **Placement config** in the **topology** (Epic 8.5): `pipeline_name -> {node, resource_pool}`. Local vs remote is a _placement decision_, not a code change. Example: face + body on the laptop iGPU; gait + SMPL on a multi-GPU box; height (cheap, calibrated) inline.
- **Multi-GPU**: a node advertises its pools (`gpu:0`, `gpu:1`); the scheduler round-robins / least-loaded across them. OpenVINO `device="GPU.0"/"GPU.1"`, ONNXRuntime CUDA `device_id`, or per-process pinning.
- **Corpus distribution**: remote nodes get the same `ActorEnrollmentEvent` NATS feed (they already subscribe) → each node has its own warm `ActorCache`. No per-call corpus shipping.

### 5. Telemetry (Epic #96 hook)

Each pipeline call wrapped in `StepTimings.span(pipeline.name)` + resource-acquire wait time; surfaced on `FrameWindow.step_timings_ms` per pipeline and logged. Makes "which model is the bottleneck on which resource" measurable — feeds the tuning loop (#51).

### 6. DAG simulation & config optimization

A discrete-event **simulator** to iterate on configuration (placement, pool sizes, batch caps, budget, model choices) **without real models or hardware** — find the optimal layout, then deploy it.

**Core principle — one scheduler, swappable clock + executor.** The simulator must NOT be a parallel reimplementation that drifts from production. Factor the scheduler (§3) over two injected dependencies:

```python
scheduler.run(window, *, clock, executor)
# prod:  clock = wall clock,     executor = real pipeline.run()  (advances real time)
# sim:   clock = virtual clock,  executor = cost-model stub      (advances simulated time)
```

The branch topo-sort, resource-pool acquire/release, batching, short-circuit, and budget logic are **identical** in both. Only _what happens inside a slot_ differs: real inference vs. "advance the virtual clock by `cost(model, batch_size, device)`." Same pattern the memory harness uses (accelerated clock). This makes sim results trustworthy and means a scheduler bug shows up in both.

**Cost model** (per pipeline × device): `cost(n) = warmup_once + dispatch + per_item·n` (batch amortization), plus `transport_ms` for remote pipelines and `compile_ms` cold-start. Seeded from real measurements already captured — e.g. OSNet iGPU steady-state, the 35.8s OpenVINO compile, body/face per-frame from `step_timings_ms`. The descriptors' `est_cost_ms` (§2) is the simulator's primary input; measured runs refine it.

**Workload model** — a scenario (reuse the memory-harness scenario DSL shape): a frame timeline with detection kinds/counts/track*ids, arrival rate, motion bursts, # concurrent people. Replay a real corpus's detection track (`fw*\*.json`) as a workload, or synthesize stress cases (10 people, 4 cameras at once).

**Resource model** — declared pools: hosts × devices (`gpu:0`, `gpu:1`, cpu), semaphore sizes, inter-host network latency. The same topology placement map (§4) drives both sim and prod.

**Outputs / what you optimize:** end-to-end window latency (p50/p95), per-pipeline wait vs run, **resource utilization**, throughput (windows/s), queue depth, budget-drop counts. A **sweep harness** runs the grid of configs and reports the Pareto frontier — **latency × cost × accuracy**. Accuracy isn't simulated: it comes from the **offline corpus probes** (`body_id_probe`, etc.) as a per-`(model, camera)` accuracy table, so the optimizer trades a cheaper/faster model against its measured accuracy on _that_ camera's geometry (e.g. body-ID 0.90 on the pool cam vs face 0.28 — pick accordingly).

**Why it pays off:** "should gait go on gpu:0 alongside face, or get its own box?" / "does batching 30 crops beat 1-at-a-time on the iGPU?" / "what budget drops p95 below 2s without losing the high-value branch?" — answered by a sweep in seconds, not by re-cabling hardware.

## New pipelines to add

| Pipeline                   | Modality   | Signal                              | resource_class | temporal             | Placement note                                                                           |
| -------------------------- | ---------- | ----------------------------------- | -------------- | -------------------- | ---------------------------------------------------------------------------------------- |
| `face_arcface` _(exists)_  | face       | face embedding                      | gpu/cpu        | no                   | durable anchor                                                                           |
| `body_id_osnet` _(exists)_ | body       | clothing appearance                 | gpu/cpu        | no                   | transient (per-visit)                                                                    |
| `ccreid_cal` **(new)**     | body_shape | clothes-invariant shape/structure   | gpu            | no                   | **durable** — survives outfit; drop-in OSNet shape                                       |
| `gait_opengait` **(new)**  | gait       | walking dynamics                    | gpu            | **yes**              | needs silhouette/pose **sequence**; strongest at distance; heavy → remote/multi-GPU node |
| `shape_smpl` **(new)**     | shape3d    | 3D body β (height/build/proportion) | gpu            | yes (or multi-frame) | direct anthropometry; heavy → remote                                                     |
| `height_calib` **(new)**   | height     | absolute height via homography      | cpu            | no                   | cheap soft signal; needs Epic 14 calibration; inline                                     |
| `plate_alpr` _(planned)_   | plate      | license text                        | gpu/cpu        | no                   | vehicles                                                                                 |

**Durability note (from 10.10):** `ccreid_cal`, `gait`, `shape3d`, `height` are clothing-_independent_ → they can be **durable KnownActor templates**, not just per-visit carry. On cameras where face fails (pool cam), gait/shape can be a _real long-term anchor_, partially substituting for face. This is the strategic payoff.

## Staged plan

- **10.11.1 — Generalize the corpus.** `EnrolledCorpus` → modality-keyed registry; pipelines read `slice(modality)`; `from_cache` generic. Migrate face/body/pet/plate to keys. No behavior change; pure pluggability. (Tests: existing router/correlation suites must pass unchanged.)
- **10.11.2 — Capability descriptors + telemetry.** Add `resource_class`/`batchable`/`temporal`/`est_cost_ms`/`placement_hint` to the Protocol (defaults preserve today's behavior) + per-pipeline `StepTimings`.
- **10.11.3 — Resource pool + batching + budget.** `ResourcePool` semaphores, cross-frame batching for batchable pipelines, window budget/timeout with logged drops.
- **10.11.3b — DAG simulator + sweep harness.** Factor the scheduler over `(clock, executor)` (small delta on .3), add the virtual-clock + cost-model executor, scenario/workload loader (replay `fw_*.json` or synthesize), and a config-sweep reporting the latency×cost×accuracy Pareto frontier. Buildable now; lets every later stage's placement be tuned _before_ the hardware exists.
- **10.11.4 — `RemotePipeline` + placement config.** RPC stub pipeline; topology placement map; multi-GPU device selection. Prove with one pipeline (e.g. a second-box OSNet) before fanning out.
- **10.11.5 — CC-ReID pipeline.** Export a CAL/AIM checkpoint to ONNX (validate like OSNet); `ccreid_cal` plugin; durable-template enrollment. Biggest robustness/effort ratio.
- **10.11.6 — Temporal pipelines (gait, SMPL).** `run_sequence` entry; silhouette (seg) / pose pre-step; OpenGait + SMPL plugins; place on GPU node.
- **10.11.7 — `height_calib`.** Homography from Epic 14 calibration; cheap inline soft signal into fusion.

Each stage is independently shippable and leaves the system working.

## Open questions

- Sequence sourcing for temporal pipelines: the rolling buffer already holds the window — `run_sequence` reads per-track frame crops from it; need a clean track→frames index.
- Remote transport: extend the FrameWindow HTTP RPC vs a dedicated identity-RPC subject on NATS. Leaning NATS (already the actor-event bus; request/reply works).
- Scheduler policy: static placement (config) first; least-loaded dynamic routing later.
- Fusion weights per modality (10.10.3) now span durable (face/gait/shape) vs transient (body) vs soft (height) — the combiner must know each modality's reliability + durability class.
