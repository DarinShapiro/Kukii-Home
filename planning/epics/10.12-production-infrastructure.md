# Epic 10.12: Production Infrastructure & Capacity Plan

**Parent epic:** Epic 10 (Identity & Recognition) · realizes the distribution design of 10.11
**Components:** topology (Epic 8.5), services/preprocessor (RemotePipeline/placement), ha-agent, infrastructure/
**Priority:** P1 — defines the hardware the platform targets + proves it meets "faster than real-time across a dozen cameras"
**Status:** planning. Capacity numbers from the DAG simulator (`scripts/dev/dag_sim.py`), seeded with measured iGPU costs + 4090-class estimates.

---

## Goal

Run identity + detection **faster than real-time across ~12 cameras**, several of which may have **simultaneous motion**, within a security alert-latency budget (~2–5 s end-to-end). Define the target hardware and how the 10.11 DAG maps onto it.

## What "real-time" means here (and why it's achievable)

Security workload is **event-driven, not continuous video analytics**:

1. **Motion-gated** (built, Epic 10.3.2): heavy models run only on cameras with active motion. A dozen cameras are mostly idle; real concurrent load = the few active _now_.
2. **Frame-sampled**: detection at **2–4 fps per active camera** is ample for "who/what is here" — not 30 fps. ~10x reduction vs. full-rate.

So the metric is **throughput (events/s) + p95 alert latency under concurrent motion**, not raw decode fps.

## Target hardware (3 tiers)

| Tier                           | Spec                                                  | Role                                                                                                                                                                                                |
| ------------------------------ | ----------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Primary GPU server**         | 16-core Threadripper, **RTX 4090**, 64 GB RAM, 10 GbE | All heavy inference: detection (yolo11x TensorRT), face (ArcFace), body/CC-ReID, pet, gait, VLM. NVDEC for camera decode.                                                                           |
| **Secondary inference server** | 8-core AMD, older AMD GPU, 32 GB RAM                  | Failover + burst spillover (RemotePipeline target); ROCm or CPU/OpenVINO fallback; can own a always-on cheap pipeline (e.g. plate).                                                                 |
| **HA Yellow (CM5 Pi)**         | Raspberry Pi CM5; **+ Coral TPU + NVMe** (planned)    | Runs the ha-agent add-on (orchestration, alert store, notify). TPU = cheap always-on motion/coarse-detect gate so the 4090 only wakes on real events. NVMe = local alert/event store + model cache. |

**Network:** all cameras on a **1 GbE PoE backplane**; primary server on **10 GbE** — ample for a dozen RTSP main streams (~5 Mbps each ≈ 60 Mbps; even 12×4K H.264 ≈ a few hundred Mbps, well under 1 GbE).

## Capacity analysis (simulator-backed)

4090 with TensorRT-class costs (detect ~7 ms, face ~8 ms, body batched ~3 ms+2/person), 4 fps detection, **one GPU pool**:

| Concurrent scenario                               | GPU-ms per 1 s wall-clock | Fraction of one 4090 |
| ------------------------------------------------- | ------------------------- | -------------------- |
| 2 cameras active, 1 person                        | 160 ms                    | **16%**              |
| 4 cameras active, 2 persons                       | 480 ms                    | **48%**              |
| 8 cameras active, 2 persons                       | 960 ms                    | **96%**              |
| **12 cameras ALL active, 3 persons (worst case)** | 1920 ms                   | **192%**             |

**Read:** at realistic concurrency (2–4 active) one 4090 runs at **16–48%** — comfortable real-time headroom. Even 8 simultaneously-active cameras fit one 4090. Only the pathological "all 12 active at once with crowds" exceeds a single GPU (~1.9x) — and that's exactly what the **secondary GPU as spillover** + the **window budget** (shed low-value branches) cover.

**Conclusion: one 4090 is the platform's workhorse with margin; the secondary GPU is failover/burst, not a requirement.** The Pi never does heavy inference.

> Caveat: numbers are simulator estimates seeded from our measured iGPU costs scaled to 4090-class. Real TensorRT yolo11x is ~7–15 ms; if higher, the knee shifts left but the headroom conclusion holds at realistic concurrency. Re-seed `dag_sim.py` with measured 4090 timings once the box exists.

## The decode wall (separate from inference)

4K **software** decode was the bottleneck on the dev box (~13 fps for ONE stream). At scale this is non-negotiable:

- **NVDEC on the 4090** decodes many 4K H.264/HEVC streams in hardware, freeing CPU + keeping the GPU fed. This is _the_ enabling fix for a dozen cameras — wire PyAV/decord to NVDEC (or run decode in the capture supervisor via FFmpeg `-hwaccel cuda`).
- Camera codec set to **H.264** (done on the pool cam) — cheaper decode than HEVC; bandwidth is a non-issue on the PoE/10 GbE backplane.
- Sub-stream for gait/coarse work (704×480, trivial to decode at 30 fps); main 4K pulled on-demand for face/plate detail.

## How the 10.11 DAG maps on

- **ResourcePool**: 4090 = one `gpu` pool (TensorRT serializes well; size 1–2 for overlap). Secondary GPU = a second named pool / remote node.
- **RemotePipeline + placement (topology)**: default everything on the 4090; spill heavy/optional pipelines (gait, SMPL, VLM) to the secondary under load or by config. Pi hosts only the TPU gate.
- **Window budget**: under the rare all-active burst, drop low-value branches (e.g. pet, gait) first, keep detection + face/body; `log()` the drop.
- **Corpus distribution**: both inference nodes subscribe to the `ActorEnrollmentEvent` NATS feed → warm `ActorCache` on each, no per-call corpus shipping.

## Staged plan

- **10.12.1 — Decode**: NVDEC-accelerated RTSP capture on the primary server; benchmark N concurrent 4K H.264 decodes.
- **10.12.2 — TensorRT detection**: export yolo11x → TensorRT (FP16/INT8) on the 4090; measure real per-frame cost; re-seed the simulator.
- **10.12.3 — Topology placement**: encode the 3-tier map (Epic 8.5); preprocessor honors `placement_hint` → local vs RemotePipeline.
- **10.12.4 — Pi TPU gate**: Coral motion/coarse-detect as the always-on wake signal; only escalate to the 4090 on a real candidate. NVMe for alert/event store.
- **10.12.5 — Secondary spillover**: RemotePipeline to the AMD box; budget-driven spill under burst; failover if primary is down.
- **10.12.6 — Soak**: 12-camera synthetic + replay load test; confirm p95 alert latency under simultaneous-motion bursts; tune pool sizes + budget from the simulator's predicted knee.

## Open questions

- AMD secondary GPU runtime: ROCm (Linux only, model-coverage gaps) vs. OpenVINO-CPU vs. Vulkan. Likely a CPU/OpenVINO fallback target unless ROCm covers our ONNX ops.
- VLM placement + cost: the VLM (grounding/explanation) is the other big GPU consumer; needs its own cost row once the backend is chosen — may justify time-slicing the 4090 or living on the secondary.
- Decode lib: PyAV+NVDEC vs. decord vs. FFmpeg-subprocess — pick on measured throughput in 10.12.1.

## Hardware saturation principle (2026-06)

**Directive:** the system must take _full_ advantage of all available
hardware — the 24-core Threadripper AND the 4090 — never leaving silicon
idle. Compute is not the constraint to economize; _idle compute_ is the
waste to eliminate. This is the active-utilization corollary of the
"never trade quality for compute" decision (§20): not only do we refuse
to degrade inputs to save compute, we actively spread work to consume the
compute we have. Quality scales up to fill the hardware.

### Three parallelism axes — and which hardware each saturates

The pipeline has three independent axes of parallelism. Saturating the
box means driving **all three at once**, matched to where each runs best:

| Axis                 | What parallelizes                                    | Best hardware                                           | Status                                                     |
| -------------------- | ---------------------------------------------------- | ------------------------------------------------------- | ---------------------------------------------------------- |
| **Frame-level**      | different frames concurrently (decode→queue→workers) | CPU cores (decode + JPEG encode release the GIL)        | built (`frame_queue.py`)                                   |
| **Branch-level**     | face / body / gait / pet on the same frame           | mixed: GPU for inference, CPU for pre/post              | built (`IdentityRouter` `asyncio.gather` + `ResourcePool`) |
| **Tile-level** (NEW) | spatial tiles of one 4K frame, batched               | GPU (batched inference) for detect; CPU for slice/merge | deferred (design below)                                    |

Tile-level is the newest and cleanest: tiles are **spatially independent**
(no ordering dependency, unlike frames), so they parallelize without the
sequencing constraint that limits frame-level. See the tiling design note
below.

### How 24 cores + a 4090 get saturated concurrently

- **CPU (24 cores)** runs everything that is _not_ GPU matmul and that
  releases the GIL or runs out-of-process: NVDEC-assisted or software
  **decode** of the dozen RTSP streams, **JPEG encode**, **tile slicing**
  - **box merge/NMS**, motion gating (MOG2), pre/post-processing for every
    model, and the orchestration. A 24-core box can keep many camera capture
    pipelines + worker pools busy in parallel — size the worker pools to the
    core count (currently a fixed 3 encode workers; on the Threadripper this
    scales to ~cores−N).
- **GPU (4090)** runs the **batched neural inference**: tiled detection
  (N tiles → one batched call), and the identity models (face/body/gait —
  already `batchable`). One `ResourcePool` `gpu` semaphore of size 1–2
  keeps the 4090 fed without thrashing; TensorRT-FP16 makes each call
  cheap enough that the GPU is the _fast_ stage and the CPU feeds it.
- **The two stay busy simultaneously** because the queue decouples them:
  CPU decodes/slices/encodes frame N+1 while the GPU runs inference on
  frame N's tiles. Neither waits on the other in steady state. That is the
  definition of saturation here — overlapped CPU prep and GPU inference,
  not one idling for the other.

### What this changes in the code (when the hardware lands)

1. **Worker-pool sizing → core-count-driven**, not the fixed 3
   (`CameraCaptureTask(encode_workers=...)` already a knob; default
   should scale to `min(cores-N, ...)` on the real box).
2. **`ResourcePool` sizes per real device** — `gpu:1-2`, `cpu:~cores`,
   tuned by the DAG simulator (10.11.3b) against _measured_ Threadripper +
   4090 costs, not laptop estimates.
3. **Tiled detection as batched GPU inference** (see below).
4. **Multi-stream concurrency** — a dozen cameras' capture tasks running
   in parallel is itself CPU-axis saturation; the per-camera independent
   buffers already support it.

### Tiling design note (the deferred full-res detection)

`detection_image_size` was raised 640→1280 as the interim fix (stop
downsampling 4K). The _proper_ full-res answer is **tiled detection**, and
it is also a parallelism win:

- **Mechanism:** slice the 4K frame into overlapping native-res tiles →
  **batch the tiles into one inference call** (NOT thread-per-tile — on
  GPU, batching is the efficient form; threads would be GIL-bound and waste
  the 4090). A distant object that's 25px in a 640-downsample is ~150px in
  its tile → detectable.
- **CPU role:** tile slicing + the box **merge/NMS across tile seams** run
  on CPU cores, overlapping the next frame's GPU inference.
- **The barrier / hard part:** merge must dedup boxes on tile overlaps AND
  assign consistent **`track_id`s across tile boundaries** (the identity
  pipeline's correlation key — `model.track()`). This is the one
  sequential fan-in after the parallel tiles, and the piece that must be
  **validated on clean footage** before shipping (cf. the CC-ReID lesson:
  no unvalidated correlation logic).
- **Tune with the simulator:** tile count, overlap, batch size, and
  per-device pool sizes are exactly the sweep the DAG sim (10.11.3b) was
  built to optimize — find the layout that saturates the 4090 without
  starving on CPU prep, _before_ committing.

**Net:** "use all the hardware" = drive frame-level (CPU), branch-level
(GPU+CPU), and tile-level (GPU-batched) parallelism concurrently, with the
queue overlapping CPU prep and GPU inference so neither idles. The
infrastructure to do this (queue, ResourcePool, router batching, the sim)
already exists; the remaining work is core-count-aware sizing + tiled
detection + simulator-tuned pool sizes against real hardware.

### The fourth axis — host-level: shape the pipeline to whatever hardware exists

The three axes above saturate _one_ box. The same decomposition extends
across **multiple servers** — and not only as the "burst spillover" the
secondary-server row (above) was originally scoped for. The pipeline is a
**graph of independent stages communicating over NATS, so every edge is a
potential partition line.** "Use all the hardware" generalizes to "cut the
graph wherever the hardware boundary is cheapest." Distribution is a
deployment choice, not a code change — the seams already exist:

| Partition point             | Mechanism (already in code)                                                          | When it's the right cut                                                                                                                  | Transport cost                                          |
| --------------------------- | ------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------- |
| **Ingestion** (per-camera)  | per-camera capture tasks + `node_id` (`config.py`, `KUKIIHOME_PREPROCESSOR_NODE_ID`) + `preprocessor_node_id` on events; `CameraConfigEvent`/`CameraConfigSubscriber` start tasks dynamically | **Decode is the bottleneck** (4K/8K NVDEC). Different cameras on different servers. | **~zero** — a camera's frames stay local through detect+crop; only the small embedding/event crosses the wire at correlation, where paths must merge anyway. |
| **Branch** (per-model)      | `placement_hint` (router.py) + RemotePipeline (10.11.4, RPC over NATS)               | One model is heavy (gait, SMPL, VLM grounding). Ship the crop, await the embedding.                                                       | crop ship + embedding return; worth it only when compute ≫ transport. |
| **Stage** (detect/id/correlate) | NATS subjects between services                                                  | Coarse split of the whole pipeline across boxes.                                                                                          | per-stage event payloads.                               |

Two consequences:

1. **Ingestion partitioning is the cleanest cut — and the one to reach for
   first.** Splitting cameras across two boxes is just two preprocessor
   instances with different `KUKIIHOME_PREPROCESSOR_CAMERAS` lists and
   different `node_id`s, both publishing to the same NATS. **No code
   change.** Crucially, it sidesteps the transport-latency caveat that
   applies to branch-level distribution: each camera's frames are fully
   independent until correlation, so all the heavy work (decode → detect →
   crop → embed) stays on one box; only the small per-track embedding
   crosses the wire. A camera is an _embarrassingly parallel_ unit.

2. **A second _high-capacity_ peer changes the secondary's role.** The
   10.12 secondary row describes a _weak_ failover box (8-core AMD, older
   GPU — "not a requirement"). A second strong server (high CPU + GPU) is a
   **true peer**, not a safety valve: you statically **partition by
   workload** and run both saturated continuously — e.g. half the cameras
   ingest+detect+identify on box A, half on box B; or detection+face/body
   on A and the heavy temporal models (gait, SMPL, VLM) on B. Both boxes
   pinned, in parallel, instead of one idling until the other overflows.

The DAG simulator (10.11.3b) already models **multi-host with inter-host
network latency**, so the placement question — "which cameras / which
models on which node?" — is answered by a sweep against latency×cost, not
a guess. `ResourcePool` already generalizes to named per-host pools
(`gpu@boxA`, `cpu@boxB`, …). The principle, stated cleanly: **the pipeline
has enough internal decomposition to shape to whatever hardware is on
hand — one box, two peers, or an edge gate + a server cluster — and the
partition seams (ingestion, branch, stage) already exist in code.**
