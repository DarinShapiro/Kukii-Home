# Epic 08.6: HA Add-on Repository Packaging

**Architecture refs:** §02, §07, docs/architecture/02-deployment-topologies.md
**Components:** ha-integration/addon, repository.json, scripts
**Priority:** P0
**Blocked by:** Epic 01, 08.5
**Blocks:** Epic 09 (HA Integration), Epic 16 #263

## Description

Add the Home Assistant **add-on repository** scaffolding so a user can paste this repo's URL into Supervisor → Settings → Add-ons → Add-on Store → ⋮ → Repositories, and SentiHome appears as an installable tile. Click Install → Supervisor builds the multi-arch image, mounts `/data/options.json`, runs the s6-overlay entrypoint, all SentiHome services start in one container managed by Supervisor.

This is the _packaging skeleton_: the add-on installs and the services boot cleanly. It does NOT make SentiHome functional inside HA — that's Epic 09 (custom integration + ha-agent code). But it does make every subsequent epic verifiable by physically installing into a real HA instance.

## Definition of done

- `repository.json` at repo root marks the repo as an add-on source
- `ha-integration/addon/sentihome/config.yaml` Supervisor manifest with options schema mirroring the Topology pydantic model
- `ha-integration/addon/sentihome/Dockerfile` builds the multi-service image on `homeassistant/<arch>-base-python` foundation
- s6-overlay service definitions in `rootfs/etc/services.d/` for each long-running SentiHome service (core, memory, ha-agent, notify, vlm-router, preprocessor)
- `run.sh` entrypoint that parses `/data/options.json` and exports it as `SENTIHOME_CONFIG` so `sentihome_shared.topology.load_topology()` picks it up
- Add-on `README.md` + `DOCS.md` + `CHANGELOG.md` rendered in the Supervisor UI
- Placeholder `icon.png` + `logo.png` (PNGs, even if just monocolor)
- CI job that lints the add-on config.yaml and builds the image for at least one arch (`amd64`)
- `docs/architecture/02-deployment-topologies.md` cross-links to the add-on install instructions
