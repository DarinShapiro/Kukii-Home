# Epic 08.5: Deployment Topology & Bootstrap Configuration

**Architecture refs:** §02, §07, infrastructure/docker/
**Components:** shared/lib-python, services/core, services/memory, services/vlm-router, services/notify, services/ha-agent, services/preprocessor, docs
**Priority:** P0
**Blocked by:** Epic 01, 02, 08
**Blocks:** Epic 09

## Description

Pre-Epic-9 prerequisite: every service currently rediscovers its dependencies via scattered env-vars (`NATS_URL`, `HA_URL`, `KUKIIHOME_ADAPTER_*`, plus hard-coded `localhost` defaults in `MemoryStoreConfig` and the vlm-router). Households deploy in materially different topologies — HA Yellow single-box, Yellow + inference NUC, fully distributed across multiple machines, or HA add-on under Supervisor — and there is no single source of truth that says "in _this_ household, NATS is at X, Postgres at Y, the VLM is at Z."

This epic adds a central **topology config**: one `kukiihome.yaml` per household (or HA add-on options.json under Supervisor), validated by a pydantic model tree, with named deployment profiles for the common shapes, `${ENV_VAR}` secret interpolation, env-var overrides on top, and a bootstrap dependency-ping that fails loud if a declared hop is unreachable.

Every existing service stops holding its own defaults and reads its slice from `load_topology()` instead. Future epics (HA, Identity, Observability, Privacy) inherit the same mechanism — no more env-var sprawl.

## Definition of done

- Single `kukiihome.yaml` schema describes deployment, bus, memory, vlm-router backends, ha-agent, notify, and NVR adapters
- Pydantic model tree with full validation (URLs, required fields, mutually-exclusive combinations)
- Three deployment profile presets: `yellow_single_box`, `yellow_plus_inference`, `distributed`
- `${ENV_VAR}` interpolation in YAML so secrets stay outside the file
- Env-var overrides via `KUKIIHOME__SECTION__SUBSECTION__FIELD=...` (double-underscore dotted-path)
- HA add-on Supervisor adapter: reads `/data/options.json` + `SUPERVISOR_TOKEN`
- `services/core`, `services/memory`, `services/vlm-router`, `services/notify`, `services/ha-agent` refactored to consume topology slices instead of env-var scatter
- `AdapterRegistry.bootstrap_from_topology()` replaces `bootstrap_from_env()`
- Bootstrap dependency-ping verifies each declared dependency (NATS connect, Postgres `SELECT 1`, Qdrant healthz, HA `/api/`, VLM `/health` or model list) and emits a single structured error report on failure
- `docs/architecture/02-high-level-architecture.md` grows a "deployment topologies" section
- `docs/onboarding.md` updated with the bootstrap config flow
- `infrastructure/docker/kukiihome.example.yaml` shipped as a copy-and-edit starting point
- All existing tests still pass; new tests cover the loader, profile merging, env-override precedence, and ping bootstrap

(labels: `epic:topology-config`, `component:shared`, `priority:p0`)
2. **feat(shared): deployment profile presets** — `yellow_single_box`, `yellow_plus_inference`, `distributed` as named dict-merge layers under the YAML. `profile: yellow_single_box` collapses every URL to in-container hostnames; `yellow_plus_inference` swaps the VLM `base_url` to a LAN IP placeholder; `distributed` is fully explicit. Selecting a profile populates defaults that the user's YAML overrides field-by-field. (labels: `epic:topology-config`, `component:shared`, `priority:p0`)
3. **feat(shared): env-var override layer** — `KUKIIHOME__VLM_ROUTER__BACKENDS__0__BASE_URL=...` style; double-underscore dotted path; numeric segments index into lists.
