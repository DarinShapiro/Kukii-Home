# Epic 10.10: Multi-Modal Identity & Fusion

**Parent epic:** Epic 10 (Identity & Recognition)
**Components:** services/preprocessor (IdentityRouter, ActorCache), services/memory (KnownActor graph), shared/lib-python (ActorEnrollmentEvent)
**Priority:** P1 — gates trustworthy "who is it" on cameras where any single modality is weak (proven: pool cam face-rec maxes at cosine 0.277 / no-match, body-ID hits 0.90)
**Blocked by:** Epic 10.4 (face), 10.5.1 (body OSNet), 10.5.3 (pet DINOv2) — all landed

---

## Problem

Face, body, and pet are enrolled and matched as **independent, parallel** modalities, each producing its own embedding and its own `ActorMatch`. Two structural gaps fall out:

1. **Enrollment can't compose.** `ActorEnrollmentEvent` is a flat record (one optional embedding per modality) and `ActorCache.upsert` was last-writer-wins per `actor_id`. Enrolling body after face for the same person *wiped* the face embedding. (Tactical merge-on-upsert shipped — see Status — but it's a stopgap, not the model.)
2. **No fusion.** A weak-but-present face and a strong body are never combined into one decision with one confidence. On a camera where face can't reach but body can (or vice versa), we throw away the signal that would have carried the identity.

Empirical driver (2026-05-29, pool cam, 50-frame corpus): same frames, same person —

| Modality | Best cosine | Match rate | Why |
| --- | --- | --- | --- |
| Face (ArcFace) | 0.277 | 0/N | ~49px face after 4K→640 downscale; no identity detail |
| Body (OSNet) | 0.989 (mean 0.899) | 11/11 ≥0.6 | full torso, large + sharp |

The right architecture must let body **carry** an identity that face (or a close camera, or manual enrollment) **anchored**.

## Key insight: face and body are not the same kind of biometric

A symmetric "store all embeddings on the actor" model is wrong because the modalities have different *temporal validity*:

| Modality | Trait type | Stability | Canonical home |
| --- | --- | --- | --- |
| **Face** (ArcFace) | Intrinsic biometric | Years | **KnownActor** — durable template |
| **Body** (OSNet) | **Appearance / clothing** | **Hours (a day)** | **Visit / SituationalContext** — transient, TTL |
| **Pet** (DINOv2) | Fur/shape descriptor | Weeks, re-confirm | KnownActor, periodic refresh |

A body embedding is "what Darin is wearing today," not "who Darin is." Storing it as a permanent `KnownActor.body_embedding` guarantees staleness. It belongs in the **transient memory layer** (VisitLedger / SituationalContext) with a short TTL, and is *derived* from a durable anchor at runtime.

## Target model

### Storage (canonical — services/memory / Neo4j)

```
Person (KnownActor)            durable identity, stable id
  └─[HAS_TEMPLATE]→ Template { id, modality, embedding,
                               quality, enrolled_at, source_cam, label, expiry? }
```

- **Many templates per modality** (face with/without glasses; body per outfit; pet per season). Matching = **max cosine over a person's templates of that modality**.
- Face/pet templates are durable; **body templates are transient** (visit-scoped, `expiry` set) — written by runtime fusion, not by enrollment.

### Wire + cache

- `ActorEnrollmentEvent` evolves from flat optional fields → **per-modality template lists**, emitted by the memory service as a **full per-actor snapshot**. With snapshots, the cache's plain overwrite is *correct* — no merge ambiguity. (Interim merge-on-upsert exists so today's partial dev publishers accumulate.)
- Preprocessor `ActorCache` stays a denormalized read-model: `actor_id → {face:[...], body:[...], pet:[...]}`.

### Runtime fusion (services/preprocessor IdentityRouter)

The body template is a **carry signal within a visit**, not a parallel identity:

1. **Anchor.** Face establishes the durable identity when a good frame / close camera permits (high precision, low recall). Manual enrollment is also an anchor.
2. **Bind + snapshot.** On identification, bind the person id to the **`track_id`** and snapshot the current body embedding as a **transient, visit-scoped body template**.
3. **Carry.** Body re-ID propagates that identity across frames and cameras for the rest of the visit — including cameras where face can't reach.
4. **Expire.** The transient body template decays / drops at visit end.

**The association key** ("these observations are the same person") is:
- **Manual** at enrollment (operator: "this is Darin"), and
- **Automatic** at runtime via **`track_id` co-occurrence** — a body on the same track as a face-identified person inherits that identity. This is what makes it scale without hand-linking every appearance.

### Confidence fusion

Per track, combine available modality similarities into one calibrated confidence. Start simple (max / noisy-OR over per-modality `sim ≥ threshold`), with per-modality reliability weights (face » body for *identity*; body » face for *this camera*). Leave a seam for a learned combiner once the feedback loop (Epic 11) produces labels.

## Staged plan

- **10.10.0 — Tactical merge (DONE).** `ActorCache.upsert` merges per-modality fields (incoming non-None overlays; None preserves). Delta semantics; can't clear via None (deactivation drops the actor). +3 tests. Unblocks a fused `darin` today.
- **10.10.1 — Template lists.** `ActorEnrollmentEvent` carries lists per modality; cache stores lists; matchers take max-over-templates. Enroll scripts append rather than clobber.
- **10.10.2 — Transient body templates.** Runtime writes visit-scoped body templates bound to `track_id` + person, with TTL/decay; router prefers durable anchors, falls back to transient carry.
- **10.10.3 — Fusion scoring.** Per-track confidence combiner across modalities + per-modality reliability weights; emit one fused `ActorMatch` with provenance (which modalities contributed).
- **10.10.4 — Memory graph.** Person—[HAS_TEMPLATE]→Template in Neo4j; memory service emits full snapshots; cache overwrite becomes canonical again.
- **10.10.5 — Cross-camera carry.** Propagate a visit identity across cameras (pool→gate→door) via shared track/appearance within a time window.

## Test strategy

- Offline corpus replay (`scripts/dev/body_id_probe.py`, `live_face_probe.py`) — already capture reusable frame corpora (`face_debug/corpus/<name>`), no walk required for reruns.
- **Separability** (the current gap): needs a second person's corpus to measure me-vs-other margin, not just self-recall.
- **Cross-session**: corpora captured on different days / outfits to quantify body-template decay and validate face-as-anchor.

## Open questions

- Snapshot vs delta as the *durable* event contract (snapshot preferred; delta only for the transient runtime writes).
- Body-template TTL: fixed (e.g. 12h) vs visit-bounded vs confidence-decay.
- Where fusion runs: preprocessor (has the embeddings + tracks) vs memory service (has history). Leaning preprocessor for the per-track decision, memory for cross-visit identity resolution.
