## Why this gets its own epic

The memory subsystem's failure modes are **cumulative and emergent** — wrong behavior is invisible at any single point in time but degrades the system over weeks. Unit tests can't catch this. We need a test harness that can:

- Drive simulated months of household events at accelerated wall-clock speed
- Verify the system's behavior against ground-truth labels
- Run the same scenario across (main) vs (PR branch) and surface trajectory differences
- Stress-test under realistic noise distributions
- Detect adversarial blind spots that humans wouldn't catch by inspection

This harness exists primarily to make **parameter tuning** safe. The Mnemosyne-adopted decay + habituation functions have 6+ hyperparameters (`a`, `b`, `d`, `c`, `Δ_max`, `t_crit`, `α_NMI`, `RS_min`). Tuning them without a regression suite is flying blind across 90-day trajectories. With the harness, you can land a tuning PR and see "Milkman scenario decay rate dropped 12%, rare-event survival went from 87% to 94%, pool-dog policy formation took 2 days less" in a single report.

## Synthetic data is the central problem

The harness needs three intertwined synthetic timelines, all temporally consistent:

```
Timeline (simulated time T₀ → T_end)
─────────────────────────────────────────────────────────────
Layer A: Camera/preprocessor events
   - motion fires + tag sets + identity matches + HA states

Layer B: Rule + policy state over time
   - initial bootstrap rules
   - user-added rules
   - VLM-authored dismissal policies (created mid-timeline)
   - TransientIntents (user-authored, TTL-bound)

Layer C: VLM responses
   - what the (mocked) VLM would have said for each invoked event,
     given the rules + policies + memory at that exact moment
```

Causal-ordering invariant: at any moment T in the timeline, Layer C's response can only depend on Layer A events ≤ T and Layer B state at T. Citations can only reference past events. Policies authored at T affect T+1 onward. This is what makes the data useful for regression testing.

## Six locked decisions

### Q1 — One canonical household, variants reserved as architectural slots

A single canonical household fixture, modeled loosely on the dev user's actual setup (front porch + driveway + garage + backyard + pool, 4 residents, 2 pets, 2 vehicles, recurring visitors). Variants (apartment, no-pets, multi-resident-with-kids, elderly + AttentionMode) are authored later only when we hit bugs the canonical household can't surface.

Fixture: `tests/synthesis/households/canonical.yaml`. Schema enforced by pydantic.

### Q2 — Four noise profiles, declared per scenario

| Profile     | Background event rate                                                          | Camera disconnects            | False detections                    |
| ----------- | ------------------------------------------------------------------------------ | ----------------------------- | ----------------------------------- |
| `none`      | 0                                                                              | 0                             | 0                                   |
| `minimal`   | ~5 events/day, deterministic placement                                         | 0                             | 0                                   |
| `moderate`  | ~30 events/day, Poisson arrivals (seeded)                                      | 1/week random                 | 1/day random                        |
| `realistic` | Full distribution: Poisson long-tail, day/night variation, weather correlation | Realistic outage distribution | Random + correlated (wind + leaves) |

Default `minimal`; tests opt in to higher fidelity. **Deterministic from `(seed, profile, household, time_window)`** — bit-identical noise across runs with same inputs. This is the bedrock for differential testing.

### Q3 — Dual VLM response streams: hand-authored oracle + recorded real-VLM

```
tests/synthesis/vlm_responses/
  oracles/                # hand-authored, deterministic; carries assertions
    canonical_milkman.yaml
    canonical_pool_dog.yaml
    ...
  recorded/               # real-VLM-captured; shared substrate with VLM eval corpus
    ...
```

Each scenario event picks its response source. Oracle for assertion-bearing tests (the milkman SHOULD get dismissed). Recorded for regression testing (after dispatcher refactor, does the same recorded response produce the same dispatcher behavior?). Recorded stream shares substrate with the VLM prompt-engineering eval corpus from task #49.

### Q4 — Adversarial subsystem is separate, Phase 2

Adversarial patterns aren't noise — they're sequenced trust-building-then-deviation trajectories that need their own generator family. Phase 2 task; not blocking Phase 1.

```
tests/synthesis/generators/adversarial/
  patterns/
    trust_build_then_deviate.py
    consistent_dismissal_then_anomaly.py
    spoofing.py
    weight_inflation.py
    rule_evasion.py
```

Each adversarial generator extends a base interface. Runs alongside normal scenarios but reported separately. Assertions verify DETECTION of the pattern, not specific outputs.

### Q5 — LLM-augmented scenario authoring, Phase 3

Defer until the canonical YAML corpus is mature (~15-20 scenarios). Then enable an LLM-generation pipeline with explicit human gating:

1. LLM prompted with canonical household + existing YAML examples → generates N candidates
2. Harness validates candidates (YAML schema, assertion consistency, truth labels, entity refs)
3. Human reviewer (approve → promote to canonical/, reject → log with reason)
4. Approved scenarios run in CI alongside hand-authored

Phase 3 task; the workflow's design is locked, the build is deferred.

### Q6 — Real-data anchoring, Phase 4 with format compatibility locked now

The scenario JSONL format must be production-anonymization-compatible **from day one**. When SentiHome has been running in production for 2-3 months:

1. Anonymization layer (consent-flow gated): production events → pseudonymized → standard-format JSONL
2. Statistical comparison tool: compare synthetic vs real distributions on event inter-arrival, tag-set frequency, KnownActor visit times, cross-camera transit times
3. Real-anchored scenarios sit alongside synthetic at `tests/scenarios/real_anchored/`

Format compatibility is what we lock now; the build is deferred.

## Phasing

Buildable now without Neo4j running. Pure-math modules + Python-dict graph mocks; integrates with real Neo4j later as a drop-in.

- `dynamics.py`: reverse-sigmoid decay, sigmoidal habituation, pruning score, redundancy score
- Unit tests for the math (H.1)

- Recorded VLM-response stream

## Definition of done (entire epic)

- Hyperparameter tuning of `(a, b, d, c, Δ_max, t_crit, α_NMI, RS_min)` can be done with confidence via the differential runner, surfacing trajectory deltas in a single per-PR HTML report
- All canonical scenarios pass in CI on every PR touching memory/retention/dynamics code
- Production data, once consented + anonymized, can plug into the harness via the standard scenario JSONL format
- Adversarial patterns we can name (trust-build-then-deviate, weight inflation, rule evasion) all have at least one scenario in the corpus that the system detects
- Visualization HTML reports let a human reviewer scan trajectory health in seconds; subtle drift that strict assertions miss is visible on charts
