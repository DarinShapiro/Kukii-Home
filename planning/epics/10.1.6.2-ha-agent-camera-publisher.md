# Epic 10.1.6.2: HA-agent → preprocessor camera-config publisher

**Parent epic:** Epic 10 (Identity & Recognition)
**Tracker:** #71
**Status:** Planning + publisher skeleton landed; credentials provider + reconciler integration deferred.

---

## Why

The preprocessor's `CameraConfigSubscriber` (Epic 10.1.6, task #69) accepts `CameraConfigEvent` broadcasts over NATS and starts/stops RTSP capture tasks dynamically. Right now the only producer of those events is `scripts/dev/publish_camera_config.py` — a manual escape hatch. To close the no-handwriting loop, the HA-side add-on (which already discovers cameras) should publish them automatically.

## What the publisher needs

For each enabled camera in `Reconciler`'s target set, publish a `CameraConfigEvent` carrying:

- `camera_id`: stable cross-camera identifier (use `DiscoverySpec.device_id`)
- `stream_url`: raw RTSP URL (`rtsp://user:pass@ip:554/path`)
- `stream_protocol`: `"rtsp"` (HLS deferred; the user explicitly rejected HLS)
- `vendor`: `"reolink"` / `"dahua"` / `"unifi"` / ...
- `sub_stream`: `True` (always use sub-stream for ingestion per Phase 10.3.1 plan)

When the Reconciler removes a camera (user disables, or auto-disables), publish `action="removed"`.

## The hard part: credentials

HA exposes camera _entities_ but not consistently the raw RTSP credentials. Three sources, in order of preference:

### Source 1: `camera.stream_source` attribute (free when it exists)

Some HA integrations populate `camera.X.stream_source` with the raw RTSP URL. Reolink in recent versions does; Dahua sometimes does. When present, this is the cheapest path: read it via the existing `HAClient` state poll.

### Source 2: HA WebSocket `camera/stream` command

The `camera/stream` WS command returns _HLS_ via HA's stream component. **Rejected per user direction** — HLS adds ~5s latency and a runtime dependency on HA in the data plane.

### Source 3: HA config-entry data

Reolink + Dahua integrations store `host`, `port`, `username`, `password` in their config entries. Accessible via the WS `config_entries/get` command **only with sufficient privilege** — add-ons running with `homeassistant_api: true` in their config can call it.

Per-vendor URL construction once credentials are obtained:

```python
REOLINK_TEMPLATES = {
    "sub":  "rtsp://{user}:{pass}@{ip}:554/h264Preview_01_sub",
    "main": "rtsp://{user}:{pass}@{ip}:554/h265Preview_01_main",
}
DAHUA_TEMPLATES = {
    "sub":  "rtsp://{user}:{pass}@{ip}:554/cam/realmonitor?channel=1&subtype=1",
    "main": "rtsp://{user}:{pass}@{ip}:554/cam/realmonitor?channel=1&subtype=0",
}
```

URL-encode the password (`%` -> `%25`, `@` -> `%40`, etc.) before substitution.

### Source 4 (fallback): operator-supplied JSON file

`/data/kukiihome/camera_rtsp_credentials.json` — one-time entry, picked up by the publisher on boot:

```json
{
  "device_id_reolink_front": {
    "ip": "192.168.68.86",
    "user": "admin",
    "password": "Kukii-Home1",
    "vendor": "reolink",
    "stream_path": "sub"
  }
}
```

This is the "no-handwriting" escape valve. The operator enters credentials _once_ in the Kukii-Home add-on UI; the publisher constructs URLs from there. **Less elegant than scraping HA config entries, but more reliable** — HA's config-entry schema is integration-private and changes between releases.

## Design: `CredentialsProvider` abstract

The publisher consumes a `CredentialsProvider` Protocol. Different implementations can be swapped without changing the publisher:

```python
class CredentialsProvider(Protocol):
    async def get_rtsp_url(
        self, *, device_id: str, vendor: str | None
    ) -> str | None: ...
```

Implementations to build (in priority order, layered):

1. **`StreamSourceAttrProvider`** — pulls `camera.X.stream_source` from HA state. Trivial to write; works for whichever vendors happen to expose it.
2. **`JsonFileProvider`** — reads `camera_rtsp_credentials.json`. Reliable fallback.
3. **`ConfigEntryProvider`** — parses HA config entries via WS `config_entries/get`. Vendor-specific; brittle. Build last.
4. **`ChainProvider`** — tries each in order, returns the first non-None.

## Reconciler integration

The publisher hooks into the Reconciler's existing diff:

```python
class Reconciler:
    async def apply(self, target_specs: list[DiscoverySpec]) -> ReconcileDiff:
        diff = self._compute_diff(target_specs)
        # ... existing start/stop logic ...
        # NEW: publish camera config events for the post-state.
        if self._camera_publisher is not None:
            for spec in diff.started:
                await self._camera_publisher.publish_configured(spec)
            for camera_id in diff.stopped:
                await self._camera_publisher.publish_removed(camera_id)
        return diff
```

The publisher is optional (`None` in tests / NATS-less environments) — same pattern the annotation cache uses on the preprocessor side.

## What this commit lands

- `camera_config_publisher.py` — the NATS publisher itself, takes a `DiscoverySpec` + credentials and produces `CameraConfigEvent`. Tested with mocked NATS.
- `CredentialsProvider` Protocol + `JsonFileProvider` implementation (the fallback). Tested.
- Unit tests for publisher correctness + credential resolution.

## What this commit does NOT land

- `StreamSourceAttrProvider` — needs a running HA + cameras to verify. Phase 10.4 starter work.
- `ConfigEntryProvider` — per-vendor scraping. Highest-risk piece; do it last, with one vendor at a time, against a real HA instance.
- Reconciler integration hook — needs careful threading through the existing reconciler test suite. Separate small commit.

## Operator flow for v0

```bash
# Inside the running Kukii-Home add-on:
#   1. Edit /data/kukiihome/camera_rtsp_credentials.json (UI later).
#   2. Restart the add-on, OR call the reconciler's reapply endpoint.
#   3. Publisher walks Reconciler's enabled specs, joins with creds,
#      publishes CameraConfigEvents on NATS.
#   4. Preprocessor's CameraConfigSubscriber spins up capture tasks.
```

## Definition of done (this epic)

- Operator can put credentials in one JSON file (or, when later providers land, in HA's config entries directly).
- Restarting the add-on rebroadcasts the full camera set.
- Toggling Enable/Disable in the UI publishes the corresponding configure / removed event within seconds.
- Zero env vars in the preprocessor's dev compose entry — the cameras list is wholly dynamic.
