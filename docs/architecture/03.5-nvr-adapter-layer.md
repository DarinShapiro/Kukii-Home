# 03.5 — NVR Adapter Layer

**Purpose:** Abstract away NVR platform dependencies so SentiHome works with any frame source — Agent DVR, Frigate, Blue Iris, Synology, QNAP, UniFi Protect, direct RTSP from cameras, or none at all. Defines the adapter contract, operating modes, and v1 vs. long-term strategy.
**Status:** drafting

---

## Why this layer exists

Early designs treated Agent DVR as a hard dependency. That assumption excludes the large installed base of Home Assistant users who already run Blue Iris, Frigate, Synology, UniFi Protect, etc. — and it locks SentiHome out of the broader HA ecosystem.

The NVR Adapter Layer makes the frame source pluggable:

- **v1:** Universal compatibility via service mode (any RTSP/ONVIF source)
- **mid-term:** Native plugins for select NVRs (latency + resource wins)
- **long-term:** NVR becomes optional as SentiHome matures (v3–v4)

This is a subtractive design philosophy. SentiHome doesn't add another layer; it makes the existing NVR layer optional over time.

---

## Operating modes (tiered performance)

Three modes cover the full spectrum of frame source integrations. SentiHome auto-detects available modes at bootstrap and selects the best one per camera.

### Mode 1: Service mode (universal — v1 default)

Separate preprocessing service consumes RTSP from any NVR (or directly from cameras), performs motion detection + frame markup + enrichment, exposes results via MCP.

```
Camera → NVR (RTSP re-broadcast) → Preprocessing Service → SentiHome
                                   ├─ Motion detection (24/7)
                                   ├─ Frame markup
                                   ├─ YOLO / face / re-ID
                                   └─ Metadata generation

Latency: ~150–370ms additional (per event, before VLM)
Resource overhead: ~1.8–2.2x (double decode, double buffers)
Compatibility: Universal (any RTSP source)
```

Works with: Agent DVR, Blue Iris, Synology, QNAP, UniFi Protect, raw cameras.

### Mode 2: Built-in mode (NVR provides preprocessing)

NVR already runs motion detection and basic enrichment; SentiHome consumes pre-enriched results.

```
Camera → Frigate (motion + YOLO built-in) → MQTT/API → SentiHome

Latency: ~50–150ms additional
Resource overhead: ~1.1–1.3x
Compatibility: NVRs with built-in AI (primarily Frigate)
```

Works with: Frigate (best fit), Synology/QNAP (basic AI only — may need supplemental).

### Mode 3: Native mode (in-process plugin)

Custom SentiHome plugin runs inside the NVR's process, with direct frame buffer access. Eliminates RTSP re-broadcast, double-decode, and most copy overhead.

```
Camera → NVR Frame Buffer → SentiHome Plugin → SentiHome (IPC/shared memory)
                            ├─ Motion detection (in-process)
                            ├─ Frame markup (in-process)
                            └─ Enrichment (in-process)

Latency: ~25–100ms additional
Resource overhead: ~1.0–1.1x (baseline + minimal)
Compatibility: NVRs with plugin/extension APIs
```

Works with: Agent DVR (planned native plugin), direct RTSP (SentiHome internal is effectively native).

```
SERVICE MODE (universal):
├─ 4 cameras:   24% CPU/GPU, 640MB RAM   ✓ OK

Same $1500 hardware:
   Native handles 16+ cameras

```

## The unified adapter contract (MCP)

Regardless of mode, SentiHome calls the same interface. The adapter implementation differs; the contract does not.

```python
class NVRAdapter(MCP):
    """
    Implemented by every adapter (service, built-in, native).
    SentiHome core never knows which mode is active.
    """

    async def list_cameras(self) -> [
        {
            "camera_id": str,
            "name": str,
            "capabilities": {
                "has_on_camera_ai": bool,
                "supported_events": [str],   # person, vehicle, motion, etc.
                "max_resolution": (w, h),
                "fps": int,
                "ptz": bool,
                "audio": bool,
            },
            "preprocessing_mode": "native" | "built-in" | "service",
            "stream_profiles": ["main", "substream"]
        }, ...
    ]

    async def get_frame_window(
        self,
        camera_id: str,
        ts_start: float,
        ts_end: float,
        with_metadata: bool = True
    ) -> {
        "frames": [frame_uri, ...],
        "metadata": {
            "motion_regions": [...],
            "detections": [...],           # YOLO bboxes if available
            "embeddings": {...},           # face, re-ID if available
            "quality_score": float,
            "preprocessing_mode": "...",
            "preprocessing_latency_ms": int
        }
    }

    async def subscribe_motion_events(
        self,
        camera_id: str,
        callback: Callable
    ) -> subscription_id
    # Push notification when motion/AI event detected

    async def enrich_frame(
        self,
        camera_id: str,
        frame_uri: str,
        models: [str] = ["yolo", "face", "reid"]
    ) -> {
        "detections": [...],
        "embeddings": {...},
        "quality_score": float
    }
    # Optional on-demand enrichment (some modes have this built into get_frame_window)

    async def get_stream_url(
        self,
        camera_id: str,
        profile: str = "main"
    ) -> str
    # Direct RTSP URL for live frame sampling (attention modes)
```

## Adapter implementations

Each platform has its own adapter that translates between platform-specific APIs and the unified contract.

### Service mode adapters (v1 priority)

**`adapter-agent-dvr-service`**
- Consumes Agent DVR OpenAPI 2.0
- Pulls clips via `/v3/clips/get`
- Subscribes to events via webhook
- Runs preprocessing in companion service
- Status: **v1 target**

**`adapter-blueiris-service`**
- Uses ha-blueiris integration for event signals
- Pulls RTSP directly from Blue Iris
- Runs preprocessing in companion service
- Status: v1 target (large user base)

**`adapter-synology-service`**
- Uses Surveillance Station Web API v3.11
- Subscribes to webhook events
- Pulls snapshots and clips via API
- Status: v1.x

**`adapter-qnap-service`**
- Uses QVR Pro OpenAPI
- Leverages frame-level API where useful
- Status: v1.x

**`adapter-unifi-service`**
- Uses official Protect API (released Feb 2026)
- RTSP streams + event webhooks
- Status: v1.x

### Built-in mode adapters

**`adapter-frigate-builtin`**
- Subscribes to Frigate MQTT topics
- Consumes Frigate REST API for clips
- Uses Frigate's YOLO detections directly
- Augments with VLM reasoning
- Status: **v1 target** (best fit for HA ecosystem)

### Direct mode

**`adapter-rtsp-direct`**
- No NVR; consumes RTSP directly from cameras
- SentiHome's internal preprocessing is essentially native (in-process)
- Handles clip archival internally (if user wants it)
- Status: **v1 target** (long-term recommended path)

### Native mode adapters (future)

**`adapter-agent-dvr-native`** (v2+)
- SentiHome plugin loaded inside Agent DVR process
- Direct frame buffer access
- IPC/shared memory to SentiHome core
- Status: Future (after v1 ships and we have real-world data)

## Mode selection and auto-detection

```python
# Bootstrap flow
async def select_preprocessing_mode(camera_config):
    """
    Auto-detect the best available mode per camera.
    SentiHome configures itself based on what's installed.
    """

    # 1. Check for native plugin
    if await native_plugin_available(camera_config.nvr):
        return "native"

    # 2. Check for NVR built-in preprocessing
    if await nvr_has_builtin_preprocessing(camera_config.nvr):
        # e.g., Frigate with MQTT enabled
        return "built-in"

    # 3. Fall back to service mode
    return "service"
```

Mode selection is **per-camera**, not per-system. A user might have:
- 2 cameras on Frigate (built-in mode)
- 2 cameras on Agent DVR (service mode, eventually native)
- 1 camera directly via RTSP (direct mode)

SentiHome handles them all through the same contract.

## On-camera AI integration

On-camera AI events (Dahua person detection, Reolink AI alerts, ONVIF events) come in through HA's camera integrations. The NVR adapter consumes them as triggers:

```
On-camera AI raises event ("person detected at 14:30:22")
  ↓
HA receives via ONVIF or vendor-specific integration
  ↓
HA normalizes to common schema:
  {
    camera_id: "front_door",
    event_type: "person_detected",
    timestamp: 14:30:22,
    confidence: 0.78,
    bbox: [...]
  }
  ↓
SentiHome receives via subscription
  ↓
SentiHome calls adapter.get_frame_window(camera_id, ts-5s, ts+10s)
  ↓
Adapter returns frames + metadata (in mode-appropriate way)
  ↓
SentiHome VLM refines/confirms/overrides on-camera opinion
```

On-camera AI is a **guide, not a gatekeeper.** Low confidence on-camera + high confidence VLM is still actioned. High confidence on-camera + VLM disagreement triggers deeper assessment.

## Service mode preprocessing service

The companion service used by all service-mode adapters. Single implementation; configured per adapter.

```
preprocessor-service (Docker container):
├─ Consumes RTSP from configured sources
├─ Motion detection (MOG2 + optical flow + size filter + temporal consistency)
├─ Frame markup (bboxes drawn on frames for VLM)
├─ Enrichment (YOLO, face, re-ID, pose, plate OCR)
├─ Metadata cache (Redis or in-memory, 5-min rolling)
└─ MCP server interface
```

Resource budget per camera (1080p @ 10fps):
- CPU: ~10–16% per core
- GPU: ~6–10% (for enrichment models)
- RAM: ~160MB (frame buffer + cache)
- Network: ~6 Mbps (RTSP re-broadcast)

Scaling guidance is preliminary (see §18 — will be refined with real deployment data).

## Long-term vision (v3–v4): NVR becomes optional

The long-term goal is to make the NVR layer optional for new SentiHome users:

```
Today (v1): "What's your NVR? We support any of them."
  ├─ Agent DVR, Frigate, Blue Iris, Synology, QNAP, UniFi Protect, or none
  └─ Service mode is universal day-1 baseline

Mid-term (v2): "Native plugins for the best experience"
  ├─ Agent DVR native plugin shipped (latency + resource wins)
  └─ Frigate built-in mode polished
  └─ Other adapters mature

Long-term (v3–v4): "You don't need an NVR"
  ├─ HA's native camera integrations + SentiHome are sufficient
  ├─ SentiHome handles motion detection, archival, clip generation
  ├─ Direct camera → HA → SentiHome is the recommended path
  ├─ NVR users still supported (existing investments preserved)
  └─ NVR is a "data source" you might have, not a layer we recommend
```

This is **subtractive design**: we're not adding more layers, we're potentially eliminating one as SentiHome's intelligence matures.

---

## Deferred decisions

These are documented as future considerations, not blocking v1:

- **SentiHome Plugin API spec** — open spec for third-party NVR vendors to implement native plugins. Defer until we have at least one native plugin shipped and battle-tested.
- **Per-tier hardware sizing guidance** — exact camera-per-hardware ratios per mode. Defer until real-world data from user's household deployment is available.
- **Performance tier marketing/communication** — how to surface "you're in service mode; native would be 2x better" in the dashboard. Defer until tiers actually differ in shipping product.
- **Shared-memory frame pipe optimization** — reduce service mode overhead via OS-level shared memory. Defer until service mode shows measurable bottleneck.
- **GPU decode pass-through** — NVIDIA NVDEC sharing between NVR and preprocessing. Defer; only relevant if GPU decode becomes the bottleneck.

## Failure modes (per §19)

NVR adapter failures are handled per §19 conventions:

- **Adapter offline** → degrade to direct RTSP if camera supports it; alert operator
- **Preprocessing service down** → fall back to NVR's native preprocessing if available; otherwise skip enrichment and let VLM work harder
- **On-camera AI event without frames** → query NVR for clip window; if NVR also down, skip this event
- **Mode-specific failures** → automatic downgrade (native → service → direct RTSP if available)

## Implementation priority (v1)

1. Define adapter MCP contract (this doc — done)
2. Build `preprocessor-service` (shared service-mode implementation)
3. Ship adapters in priority order:
   - `adapter-rtsp-direct` (simplest, no external deps)
   - `adapter-agent-dvr-service` (the original design choice)
   - `adapter-frigate-builtin` (best HA ecosystem fit)
   - `adapter-blueiris-service` (biggest external user base)
   - `adapter-synology-service`, `adapter-qnap-service`, `adapter-unifi-service` (as demand drives)
4. Auto-detection logic at bootstrap
5. Mode + latency observability in dashboard

Native modes, Plugin API spec, and shared-memory optimizations are explicitly **out of v1 scope**.
