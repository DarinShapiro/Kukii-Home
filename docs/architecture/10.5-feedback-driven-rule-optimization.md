# 10.5 — Feedback-Driven Rule Optimization & Replay Testing

**Purpose:** How the system automatically learns and improves rules through user feedback, variant testing against historical video, and safe rollout of optimized configurations.
**Status:** drafting

---

## Core principle: User provides ground truth; system optimizes

The user's job is simple: **indicate when the system made a mistake.**

```
User feedback signals:
  • "You missed this" (false negative)
  • "That was wrong" (false positive)
  • "That's correct" (positive feedback, reinforce)
```

The system's job is to:
1. Analyze conditions when the mistake occurred
2. Generate candidate optimization strategies (variants)
3. Test variants against the historical clip
4. Identify winning configurations
5. Gradually deploy them to future similar events
6. Track whether they actually improved performance

**The user never chooses a strategy.** The system figures it out automatically.

## Feedback loop trigger

Feedback can come from:

### 1. Post-alert dismissal (§15 alerting)

```
Alert fires: "Unknown person at door"
User clicks: [✗ False alarm] [? Not sure] [✓ Correct]

If false alarm:
  → Escalate to optimization
  → "Help me understand: what was wrong?"
  → User provides context (optional)
    - "That was a delivery driver, not a stranger"
    - "It was just a shadow from a tree"
    - "I don't care about this, suppress it"
```

### 2. User-reported misses

```
User in app → History view → "This event was missed"
  Or: "Why didn't you alert me about [clip]?"
  
System retrieves that clip → triggers optimization
```

### 3. Pattern mining (automatic)

```
System observes repeated dismissals of same rule type:
  • Tuesday mornings, always dismissed
  • User dismisses "motion detected" 5 times in a row
  
Pattern → hypothesis: "Rule is too sensitive during this time/context"
  → Auto-flag for optimization (user can confirm or dismiss)
```

## Variant generation strategy

When a misfire is detected, the system generates N candidate configurations that might have fixed it.

### Generation approach

```
Misfire analysis:
  1. What were the conditions?
     - Time of day / lighting
     - Scene complexity (1 subject? many?)
     - Confidence levels of original detections
     - Frame sampling: was the moment captured?
     - Preprocessing used: was enrichment sufficient?
  
  2. What could have fixed it?
     - Lighting issue? → try preprocessing (background subtraction, contrast enhance)
     - Small subject? → try higher resolution or denser sampling
     - Low confidence? → try higher compute level
     - Multiple subjects confusing the model? → try annotated preprocessing
     - Frame sampling missed the moment? → try adaptive/denser interval
  
  3. Generate combinations (combinatorial or guided):
     - Sample 10–20 variant configurations
     - Explore 2–3 parameter changes per variant
     - Avoid extreme combinations (e.g., max resolution + max fps + high-compute is overkill)
```

### Available optimization knobs

**Inference / Compute:**
- `compute_level`: low | med | high
  - low: fast local inference (YOLO small, simple VLM)
  - med: balanced (standard YOLO, local VLM)
  - high: accurate but slower (YOLO large, high-effort VLM reasoning)

- `vlm_model`: local_fast | local_accurate | cloud_fallback
- `detector_model`: yolov5_small | yolov5_medium | yolov5_large | rt_detr

**Frame Selection:**
- `frame_resolution`: 480p | 720p | 1080p | 2560p (higher = better detail, slower)
- `frame_interval`: 0.5s | 1s | 2s | 4s (denser = more context, more compute)
- `adaptive_interval`: false | true (increase interval as sequence lengthens)
- `frame_budget`: 4 | 8 | 16 | 32 (total frames per event)
- `motion_triggered`: false | true (only sample on scene change)

**Preprocessing (OpenCV-style):**
- `preprocessing_strategy`: 
  - `none` — raw frames
  - `background_subtraction` — isolate moving subjects (GMG, KNN, MOG2)
  - `edge_detection` — emphasize boundaries (Canny, Sobel)
  - `morphological` — shape enhancement (dilate/erode)
  - `contrast_enhance` — boost low-light visibility
  - `histogram_equalize` — normalize exposure
  - `composite` — combine multiple (e.g., background sub + edge detect)

**Enrichment Density:**
- `enrichment_level`:
  - `minimal` — just class + confidence
  - `standard` — bounding boxes, confidence, track ID
  - `annotated` — + pose keypoints, semantic labels, relationships
  - `heavy` — + identity annotations, attribute details, all subjects marked

**Other:**
- `night_mode`: false | true (use night-vision preprocessing if available)
- `multi_subject_mode`: false | true (annotate all subjects, not just primary)
- `crop_strategy`: none | tight_zoom | multi_crop (send crops of small subjects)

### Variant example

```
Original rule: "Alert if dog in front yard"
  Failure: Monday 22:00 low-light, dog missed

Generated variants:
  Variant A:

  ... (10–20 variants total)
```

## Replay testing framework

### Archival requirement

Every event must be archived with:

```json
{
  "event_id": "camera-doorbell-20260523T220712-track42",
  "clip_uri": "s3://clips/doorbell/20260523T220712_track42.mp4",
  "frames": [
    { "frame_id": 0, "ts": "22:07:12", "uri": "s3://frames/...", "raw": true },
    { "frame_id": 1, "ts": "22:07:13", "uri": "s3://frames/...", "raw": true },
    ...
  ],
  
  "original_analysis": {
    "detections": { ... },
    "enrichment": { ... },
    "vlm_output": { ... },
    "rule_fired": ["rule_dog_alert"],
    "action": "notify"
  },
  
  "user_feedback": {
    "ts": "2026-05-23T22:15:00Z",
    "feedback_type": "missed | false_alarm | correct",
    "context": "low light, dog at side of yard",
    "severity": "high | medium | low"
  }
}
```

### Replay engine

When optimization is triggered, replay the hot path with different configs:

```python
# Pseudocode
def replay_variant(variant_config, archived_event):
    """
    Re-run detection + VLM analysis with variant config
    against archived frames.
    """
    
    # Re-sample frames according to variant
    frames = sample_frames(
        archived_event.frames,
        interval=variant_config.frame_interval,
        resolution=variant_config.frame_resolution
    )
    
    # Apply preprocessing
    frames = preprocess(frames, variant_config.preprocessing_strategy)
    
    # Run detector with variant's model
    detections = detector.enrich(
        frames,
        model=variant_config.detector_model,
        enrichment_level=variant_config.enrichment_level
    )
    
    # Run VLM with variant's compute level
    vlm_output = vlm.analyze(
        frames,
        detections,
        compute_level=variant_config.compute_level,
        model=variant_config.vlm_model
    )
    
    # Apply rule
    rule_result = evaluate_rule(vlm_output, archived_event.rule_fired[0])
    
    return {
        "variant_id": variant_config.id,
        "detected": rule_result.alert_fired,
        "confidence": rule_result.confidence,
        "detections_found": len(detections),
        "compute_time_ms": vlm_output.latency_ms,
        "compute_cost": estimate_cost(variant_config)
    }

# Test all variants against the historical clip
results = []
for variant in variants:
    result = replay_variant(variant, archived_event)
    results.append(result)
```

### Metrics & ranking

For each variant, compute:

**On the failure case (historical clip):**
- `catches_miss`: did it detect what was originally missed? (boolean)
- `confidence`: what confidence did it report? (0.0–1.0)
- `false_positives`: did it fire on irrelevant things in that clip?
- `compute_time_ms`: how long did it take?
- `resource_cost`: GPU memory, tokens for VLM, etc.

**On related historical clips (same rule, similar context):**
- `tp_count`: true positives on past similar events
- `fp_count`: false positives on past similar events
- `fn_count`: false negatives on past similar events (should decrease)
- `precision`: TP / (TP + FP)
- `recall`: TP / (TP + FN)
- `f1_score`: harmonic mean

**Cross-check on historical false positives:**
```
If original rule fired on false positive at Monday 14:00,
does Variant B still fire?
  If yes: Variant B has same false positive (worse if more sensitive)
  If no: Variant B is more specific (good!)
```

### Variant ranking

```
Ranking criteria (tunable per scenario):
  1. Primary goal: does it catch the original miss?
  2. Secondary: what's the precision on similar events?
  3. Tertiary: resource cost (prefer lower)
  
Example ranking:

Variant | Catches Miss | Precision | Recall | F1    | Compute(ms) | Cost  | Score
--------|-------------|-----------|--------|-------|-------------|-------|-------
A       | ✓ Yes       | 0.92      | 0.88   | 0.90  | 450         | med   | 0.92
B       | ✓ Yes       | 0.85      | 0.94   | 0.89  | 320         | low   | 0.88
C       | ✗ No        | 0.88      | 0.82   | 0.85  | 200         | low   | 0.65
D       | ✓ Yes       | 0.79      | 0.91   | 0.84  | 600         | high  | 0.75
...

Top candidates:
  1. Variant A (best overall; catches miss, good precision/recall, reasonable cost)
  2. Variant B (slightly lower precision but catches more, cheaper)
```

## Conflict detection

When a variant is optimized for one scenario, check that it doesn't break others.

### Cross-scenario validation

```
Original rule: "Alert if dog in front yard"
Optimized for: Monday 22:00 miss (low-light scenario)
Winning variant: high-compute + contrast-enhance + 1080p

Check against:
  1. Daytime false positives (does it increase FP in daylight?)
  2. Other dog-related scenarios (shadows, leaves, reflections)
  3. Non-dog alerts (is the rule still specific to dogs?)
  
If Variant increases false positives elsewhere:
  → Mark as "low-light-specific" variant (only use when conditions match)
  → Or blend: use variant only between 20:00–06:00
```

## Safe rollout & graduation

Variants don't replace rules immediately. They're tested gradually.

### Phased deployment

```
Phase 1: Silent testing (0–7 days)
  Variant A runs in parallel on new events
  No user impact; metrics collected only
  Check: does it improve metrics on new similar events?

Phase 2: Shadow mode (7–14 days)
  Variant A results logged; shown in telemetry
  If metrics are stable/improving: advance to Phase 3
  If metrics degrade: archive variant

Phase 3: Gradual rollout (14–30 days)
  - Day 14: apply variant to 10% of events (similar context)
  - Day 18: apply to 25% of events
  - Day 25: apply to 50% of events
  - Day 30: apply to 100% (variant becomes new rule)
  
  At each step, monitor:
    - True positive rate
    - False positive rate
    - User dismissals / feedback
  
  If metrics diverge: rollback to previous step

Phase 4: Replacement (day 30+)
  Original rule replaced by optimized variant
  Variant config merged into rule definition
  Old rule archived (for rollback if needed)
```

### Rollback triggers

Automatic rollback if:
- FP rate increases > 20% compared to original
- FN rate increases (fewer catches, not more)
- User dismissals spike (5% of events dismissed suddenly)
- Compute cost exceeds budget (variant too expensive to deploy widely)

```
Example rollback:
  Day 22: Variant A shows 15% increase in FP
  → Alert operator
  → Monitor for 48h
  → If FP continues high: auto-rollback to original rule
  → Archive variant with note: "too many false positives in daylight"
```

## System requirements

For this to work, the system needs:

1. **Event archival** — every event clip + frames + metadata stored
2. **Replay engine** — ability to re-run detection + VLM pipeline with different configs
3. **Variant generator** — combinatorial or guided exploration of parameter space
4. **Metrics pipeline** — compute TP/FP/FN/precision/recall on historical windows
5. **Safe rollout framework** — phased deployment with rollback triggers
6. **User dashboard** — visibility into what's being tested and why
7. **Feedback collection** — easy way for users to report misses/false alarms
8. **Storage/retention** — keep clips long enough to replay (30+ days recommended)
